# S2 产业验证智能体任务

> 本文件是 S2 智能体的研究任务书。S2 不是纯代码模块：代码负责计算，智能体负责发现、验证和分流产业事件。
> 每日完整执行规程见 `docs/daily_update_runbook.md`。

## 一、任务目标

每天发现可能影响 159567 创新药 ETF 产业预期的事件和线索，并把它们分成三类：

| 类别 | 去向 | 是否进入 S2 正式评分 |
| --- | --- | --- |
| 已验证重大事件 | `s2/data/*.csv` 正式事件库 | 是 |
| 未达阈值或待验证线索 | `s2/output/watchlist.md` | 否 |
| 已排除线索 | agent run 或 watchlist 历史关闭区 | 否 |

核心原则：**watchlist 可以松，正式事件库必须严**。不得把未验证传闻、单一媒体线索、低相关事件直接写入正式事件库。

## 二、每日输入

执行 S2 智能体扫描前，至少读取：

| 输入 | 用途 |
| --- | --- |
| `docs/daily_report.md` | 查看 S1 当日状态、资金/价格/广度是否异常 |
| `s2/output/reports/YYYY-MM-DD.md` 或最新 S2 报告 | 查看当前 S2 评分、事件库状态和待验证项 |
| `s3/output/ai_style_daily_report.md` | 判断当日是否可能由 AI/科技成长风格解释 |
| `s2/output/watchlist.md` | 回看未关闭线索，判断升级、延续、排除或过期 |
| `s2/data/*.csv` | 避免重复入库，确认历史事件背景 |

如果当天正式报告还未生成，可先读取最近一日输出和行情数据，完成事件扫描后再运行日报流程。

## 三、今日研究问题

每天不能只问“有没有公告”。执行者必须先形成 2-5 个今日研究问题，例如：

- 159567 相对 159557、589720 或 AI_CORE 的表现是否异常？
- 今日价格变化能否由正式产业事件解释，还是更像资金/风格波动？
- watchlist 中有没有线索需要升级、删除或继续观察？
- 是否有 BD、临床、监管、政策、海外竞品事件会改变创新药预期？
- 是否有公司级事件对 159567 权重标的或行业 beta 有外溢影响？

这些问题要写入当天 `s2/output/agent_runs/YYYY-MM-DD.md`。

### 事件类型

| 类型 | 进入正式事件库的最低要求 |
| --- | --- |
| BD 交易 | license-in/out、战略合作、商业化许可；首付款 > 5000 万美元或总金额/标的影响足够重大 |

### 来源分级

| Tier | 来源 | 用法 |
| --- | --- | --- |
| Tier 1 | 公司公告/IR、交易所公告、NMPA/CDE、FDA、正式会议摘要 | 可作为正式入库主证据 |
| Tier 2 | FierceBiotech、Endpoints、STAT、医药魔方、Insight 等权威行业媒体 | 可作为辅助证据，重大事件需尽量回到 Tier 1 |
| Tier 3 | 通用新闻、社媒、券商评论、论坛转述 | 只能进入 watchlist 或 rejected，不得单独入库 |

最低覆盖不是上限。若今日行情、S3 风格、watchlist 或行业会议触发异常，执行者必须自主扩展搜索方向。

## 五、候选事件 Triage

每个候选事件都要做分流判断：

| 状态 | 含义 | 处理 |
| --- | --- | --- |
| `promote` | 已验证且达到入库阈值 | 写入 `s2/data/*.csv`，在 agent run 写明去重依据 |
| `watchlist` | 相关但未达正式阈值 | 更新 `s2/output/watchlist.md` |
| `needs_verify` | 有线索但来源不足 | 更新 watchlist，并标明下一步验证路径 |
| `reject` | 不相关、重复、低重要性或无法验证 | 在 agent run 说明排除理由 |
| `expire` | 旧线索超过观察期且无进展 | 移入 watchlist 历史关闭区 |

判断维度至少包括：

- 来源可靠性。
- 是否有明确日期、主体、事件类型。
- 是否影响 159567 相关标的或创新药行业预期。
- 是否达到正式入库阈值。
- 是否与历史事件库重复。
- 是否需要后续交易日或公告验证。

## 六、正式入库规则

正式事件库文件：

| 事件类型 | 文件 |
| --- | --- |
| BD 交易 | `s2/data/bd_events.csv` |
| 临床数据 | `s2/data/clinical_events.csv` |
| 监管审批 | `s2/data/regulatory_events.csv` |
| 业绩事件 | `s2/data/earnings_events.csv` |
| 政策风险 | `s2/data/policy_risk_events.csv` |

写入必须使用：

```python
from s2.event_store import append_events
append_events(path, [event_dict])
```

入库前必须确认字段完整、来源可追溯、没有重复。入库后运行报告生成命令，确认事件已正确显示。

## 七、Watchlist 规则

`s2/output/watchlist.md` 是研究线索池，不是正式事件库。它用于承接：

- 未达正式阈值但值得继续看的一手线索。
- 需要交叉验证的媒体报道。
- 与当天行情相关但产业证据不足的解释假设。
- 即将召开的会议、PDUFA、CDE 关键节点、财报窗口等。
- 旧线索的升级、延续、排除或过期记录。

watchlist 不得被用于 S2 正式评分。若未来需要在报告中展示，只能以“观察线索，不参与评分”的形式出现。

## 八、每日输出

无论是否新增正式事件，每天必须写：

```text
s2/output/agent_runs/YYYY-MM-DD.md
```

运行记录至少包括：

1. 今日研究问题。
2. 市场与报告线索。
3. watchlist 回看。
4. 最低覆盖来源。
5. 自主扩展搜索。
6. 候选事件 triage。
7. 正式写入事件库情况。
8. watchlist 更新。
9. 对今日 S2 结论的影响。
10. 验证命令和结果。

如果没有新增正式事件，不能只写“无新增”。必须说明检查范围、候选排除理由、watchlist 是否变化。


若执行完整日报流程，则按 `docs/daily_update_runbook.md` 运行统一入口并审计 S1/S2/S3 日期与数据源。
