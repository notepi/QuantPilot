# S2 事件收集智能体任务

> 本文档将 `docs/data_governance_plan.md` 第十一章的智能体任务设计固化为可执行任务。
> 维护入口：每日由智能体执行，补充事件库。

每日完整执行规程见 `docs/daily_update_runbook.md`。执行者不能只运行 `s2.daily_report_flow` 后结束；必须先完成本文件定义的事件扫描，并写入 `s2/output/agent_runs/YYYY-MM-DD.md` 运行记录。

## 目标

每天发现并记录可能影响 159567 创新药 ETF 的事件，写入 `s2/data/` 目录下的事件库。

### 关键事件类型

1. **BD 交易**：license-in/out、战略合作、商业化许可
2. **临床数据**：Phase II/III 数据读出、会议报告（ASCO、ESMO 等）
3. **监管审批**：NDA/BLA 受理、获批、CDE 公示
4. **政策风险**：BIOSECURE、医保目录调整

### 重要性判断

| 事件类型 | 重要性阈值 |
|---------|-----------|
| BD 交易 | 首付款 > 5000 万美元值得记录 |
| 临床数据 | Phase II 以上，主要终点达到 |
| 监管审批 | NDA/BLA 级别 |
| 政策风险 | 可能影响整个行业 |

### 来源可靠性

| Tier | 来源 |
|------|------|
| Tier 1 | 公司官方 IR、NMPA、FDA 公告 |
| Tier 2 | FierceBiotech、Endpoints News |
| Tier 3 | 其他媒体报道 |

## 你需要自己决定

1. **去哪里找信息** — 公司官网 IR 页面、行业媒体、监管机构公告
2. **什么信息值得记录** — 重要性判断 + 相关性判断
3. **如何验证信息可靠性** — 交叉验证、来源可靠性评估
4. **如何组织数据** — 字段填充、去重逻辑

## 写入接口

```python
from s2.event_store import append_events
append_events(path, [event_dict])
```

## 验证方式

完成后运行：

```bash
uv run python -m s2.generate_s2_report
```

检查事件是否正确显示在报告中。

## 运行记录要求

无论今日是否新增事件，都必须创建：

```text
s2/output/agent_runs/YYYY-MM-DD.md
```

记录内容至少包括：

- 今日新增事件数量。
- 检查过的来源类型。
- 候选事件及处理结论。
- 写入了哪些事件库文件。
- 验证命令和结果。

如果没有新增事件，必须写明“今日无新增重大产业事件”，并说明已检查的来源范围。
